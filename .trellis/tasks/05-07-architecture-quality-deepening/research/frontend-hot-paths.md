# Research: frontend hot paths

- Query: Scopy frontend architecture deepening opportunities for performance, stability, and maintainability after v0.7.6 row descriptor, thumbnail scheduler, and frontend profile isolation work.
- Scope: internal
- Date: 2026-05-07

## Findings

### Source Context

The current task goal is architecture deepening, not a broad feature rewrite. The task PRD requires at least three evidence-backed candidates, a ranked recommendation, and later implementation/check work with tests and performance gates when hot paths are affected (`.trellis/tasks/05-07-architecture-quality-deepening/prd.md:1`). Current release metadata is `v0.7.6` and records passed build/unit/strict gates plus a standard frontend profile guard (`doc/meta/release-current.yml:1`). The v0.7.6 profile says the release moved row-ready display data into `HistoryItemRowDescriptor`, routed visible row thumbnails through `HistoryRowThumbnailLifecycleScheduler`, and isolated frontend profile app processes (`doc/perf/release-profiles/v0.7.6-profile.md:14`). It also warns that the standard profile is a regression guard, not a clean universal before/after UI win (`doc/perf/release-profiles/v0.7.6-profile.md:24`).

Product constraints are unchanged: Scopy must preserve durable history, direct item actions, hover preview/export/annotation, incremental loading, thumbnails, settings, and realistic-snapshot responsiveness (`doc/current/product-spec.md:23`, `doc/current/product-spec.md:29`, `doc/current/product-spec.md:43`, `doc/current/product-spec.md:55`, `doc/current/product-spec.md:115`). The active architecture doc keeps preview/export as UI-shell responsibilities but requires stored content to remain source-of-truth input, not a side channel that mutates persistence (`doc/current/architecture.md:36`). Operational invariants require heavy thumbnail/file/export work to stay backgrounded and bounded (`doc/current/architecture.md:47`) and treat explicit test surfaces as architecture, not just implementation style (`doc/current/architecture.md:55`).

### Code Patterns

The v0.7.6 work is real and should not be re-proposed as unfinished. `HistoryItemRowDescriptor` centralizes row-ready display values and records `row.display_model_ms` within that Module (`Scopy/Presentation/HistoryItemRowDescriptor.swift:40`). The thumbnail scheduler has an explicit dependency Interface for cache/load/scroll/sleep/cancel and returns path-tagged commit results (`Scopy/Views/History/HistoryRowThumbnailLifecycleScheduler.swift:17`, `Scopy/Views/History/HistoryRowThumbnailLifecycleScheduler.swift:56`). The row thumbnail views now use the scheduler as a deep Module while retaining view-owned `@State` and `.task(id:)` cancellation (`Scopy/Views/History/HistoryItemThumbnailView.swift:10`, `Scopy/Views/History/HistoryItemThumbnailView.swift:30`, `Scopy/Views/History/HistoryItemFileThumbnailView.swift:77`).

The remaining largest shallow Module is `HistoryItemView`. Its Interface includes item data, selection state, settings, eight action closures, image data loading, shared WebView, interaction coordinator, three popover booleans, popover request closure, and dismiss closure (`Scopy/Views/History/HistoryItemView.swift:59`). Internally it also owns several workflows that are not exercised through a single testable Interface: markdown file preview cache/freshness/rendering (`Scopy/Views/History/HistoryItemView.swift:349`), row hover dispatch (`Scopy/Views/History/HistoryItemView.swift:932`), image preview prefetch and downsample (`Scopy/Views/History/HistoryItemView.swift:1037`), file preview prefetch with image/video/QuickLook branches (`Scopy/Views/History/HistoryItemView.swift:1119`), text markdown detection/rendering (`Scopy/Views/History/HistoryItemView.swift:1483`), and stable markdown metric normalization (`Scopy/Views/History/HistoryItemView.swift:982`).

Markdown preview metric normalization appears in multiple places instead of one deep Module. Cached file-preview metrics are normalized with fallback width at `HistoryItemView.swift:369`, newly fetched file-preview cached HTML metrics normalize the same shape at `HistoryItemView.swift:473`, and text pre-measurement normalizes the same fallback-width checks at `HistoryItemView.swift:1003`. The `MarkdownPreviewCache` Interface stores HTML/metrics/file preview entries (`Scopy/Views/History/MarkdownPreviewCache.swift:48`, `Scopy/Views/History/MarkdownPreviewCache.swift:65`) but does not own the stability policy, so callers must know the same invariants.

The list popover state seam is half-deep. `HistoryListView` centralizes one-active-popover state and shared WebView detach/reopen behavior (`Scopy/Views/HistoryListView.swift:169`, `Scopy/Views/HistoryListView.swift:222`), while row views still own image/text/file popover bindings, system-dismiss handling, scroll-wheel dismissal, and task state transitions (`Scopy/Views/History/HistoryItemView.swift:793`, `Scopy/Views/History/HistoryItemView.swift:820`, `Scopy/Views/History/HistoryItemView.swift:831`, `Scopy/Views/History/HistoryItemView.swift:919`). This is not necessarily wrong, but popover correctness spans two Modules and UI tests instead of one direct Interface.

The scroll/pointer interaction seam is useful but time-bound and not fully injectable. `HistoryListInteractionCoordinator.isHoverPreviewSuppressed` reads `CFAbsoluteTimeGetCurrent()` directly to enforce its cooldown (`Scopy/Views/History/HistoryListInteractionCoordinator.swift:19`). Tests wait a real 300ms to clear suppression (`ScopyTests/HistoryListInteractionCoordinatorTests.swift:28`). `ListLiveScrollObserverView.ObserverView` also owns `NSScrollView` discovery, notification attachment, and mouse event monitoring in one AppKit Adapter (`Scopy/Views/History/ListLiveScrollObserverView.swift:61`, `Scopy/Views/History/ListLiveScrollObserverView.swift:123`, `Scopy/Views/History/ListLiveScrollObserverView.swift:172`). This is a stability/Locality candidate, but not the strongest performance candidate because the current profile shows row/thumbnail buckets are already small relative to main-runloop activity.

The latest v0.7.6 profile shows why row micro-optimization should not be first. In the standard run, `row.display_model_ms.p95` is sub-millisecond across real-snapshot scenarios, while `image.thumbnail_load_total_ms.p95` already improved under the scheduler (`doc/perf/release-profiles/v0.7.6-profile.md:30`). Profile-level frame/drop counters are explicitly noisy (`doc/perf/release-profiles/v0.7.6-profile.md:41`). Long-frame attribution in the raw summary still shows main-thread coverage around 84.8%-97.3% while app-attributed coverage remains low (`logs/perf-frontend-profile-2026-05-07_03-22-16/frontend-scroll-profile-summary.md:80`). The older failed row-id experiment also worsened `row.file_preview_ms.p95` and main-runloop p95, confirming that row-local identity churn is not a high-confidence path (`logs/perf-frontend-profile-lowpower-mixed-row-id-removed-2026-05-02_02-23-29/frontend-scroll-profile-summary.md:18`, `logs/perf-frontend-profile-lowpower-mixed-row-id-removed-2026-05-02_02-23-29/frontend-scroll-profile-summary.md:23`).

## Candidates

### 1. Deepen Hover Preview Loading Into A `HistoryHoverPreviewPipeline` Module

**Rank:** 1. Most suitable first implementation for this task.

**Files / Modules:** `HistoryItemView`, `HistoryItemPreviewCoordinator`, `HoverPreviewModel`, `HoverPreviewLoader`, `HoverPreviewImageCache`, `MarkdownPreviewCache`, `HistoryItemImagePreviewView`, `HistoryItemTextPreviewView`, `HistoryItemFilePreviewView`, and focused tests under `ScopyTests`.

**Problem:** The current hover preview Module is shallow at the wrong place. `HistoryItemPreviewCoordinator` exposes a small Interface, but it mostly owns tokens and task cancellation (`Scopy/Views/History/HistoryItemPreviewCoordinator.swift:13`, `Scopy/Views/History/HistoryItemPreviewCoordinator.swift:91`). The real Implementation is still spread through `HistoryItemView`: image preview prefetch uses delay, screen scale, cache keys, storage-vs-data loading, downsampling, cache writes, and cancellation checks (`Scopy/Views/History/HistoryItemView.swift:1037`); file preview repeats the same shape with image/video/QuickLook branches (`Scopy/Views/History/HistoryItemView.swift:1119`); markdown file preview owns TTL, stale-cache fallback, file IO, HTML rendering, metrics preservation, and cache writes (`Scopy/Views/History/HistoryItemView.swift:349`); text preview owns markdown detection/render/cache state (`Scopy/Views/History/HistoryItemView.swift:1483`). The row view is therefore a caller and an implementation container at the same seam.

**Solution:** Create a deeper hover-preview pipeline Module whose Interface is closer to `start preview for item plus descriptor plus policy plus callbacks`, and whose Implementation owns delay, budgeted work, cache key planning, image/file/text/markdown loading, cancellation guard points, and metric names. Keep SwiftUI `popover` presentation and `HoverPreviewModel` mutation on the row side or through a narrow main-actor result Adapter. Internal Adapters should include image data loading, file preview loading, markdown renderer/cache, image cache, screen metrics, sleep/cancel, and metric recording so unit tests can exercise behavior without UI. This should be a behavioral extraction, not a product change.

**Benefits:** Depth improves because callers no longer need to know every preview path cache key, delay, file kind, markdown threshold, and metric ordering. Leverage improves because image/file/text/markdown preview tests can hit one Interface rather than individual view-private functions. Locality improves because future preview bugs, cache invalidation changes, or metric policy changes live in one Module rather than inside a 1843-line row. It also makes main-thread risk more visible: the pipeline can deliberately return render-ready results to the row rather than interleaving `@MainActor` state mutation with detached work.

**Risks:** This touches a fragile UI workflow: hover delay, scroll suppression, popover re-open behavior, shared WebView attachment, and UI-test tap-preview mode must stay unchanged. If the Interface becomes too generic, it could become another shallow wrapper. Keep the first implementation narrow: preserve existing `HoverPreviewPopoverKind`, `HoverPreviewModel`, cache names, metric names, and popover rendering views.

**Tests / Verification:** Add unit tests for image preview cache hit/miss, file image/video/QuickLook branch selection, markdown file stale-cache fallback, text markdown detection cache use, cancellation before commit, and metric emission via injected Adapters. Keep existing `HistoryItemPreviewCoordinatorTests`, `HoverPreviewImageCacheTests`, `HoverPreviewImageQualityPolicyTests`, `HoverPreviewTextSizingTests`, and add a focused `HistoryHoverPreviewPipelineTests`. Run `make build`, `make test-unit`, `make test-strict`. Because hover preview affects scroll and preview hot paths, also run `xcodebuild test -project Scopy.xcodeproj -scheme Scopy -destination 'platform=macOS' -only-testing:ScopyUITests/HistoryListUITests/testHoverPreviewDismissesOnScroll` and at least `make perf-frontend-profile`; use `make perf-frontend-profile-standard` before claiming performance confidence.

### 2. Extract Markdown Preview Metrics Normalization Into A Deep Cache/Policy Module

**Rank:** 2. Smaller and safer than candidate 1, but less leverage.

**Files / Modules:** `HistoryItemView`, `MarkdownPreviewCache`, `HoverPreviewTextSizing`, `MarkdownPreviewMeasurer`, `HistoryItemTextPreviewView`, `HistoryItemFilePreviewView`, and `HoverPreviewTextSizingTests`.

**Problem:** The current `MarkdownPreviewCache` Interface stores and updates raw HTML/metrics/file-preview entries (`Scopy/Views/History/MarkdownPreviewCache.swift:48`, `Scopy/Views/History/MarkdownPreviewCache.swift:61`, `Scopy/Views/History/MarkdownPreviewCache.swift:76`), but every caller must know how to turn measured WebKit metrics into stable popover dimensions. The same fallback-width rules appear in cached markdown file preview, newly fetched markdown file preview, and text pre-measurement (`Scopy/Views/History/HistoryItemView.swift:369`, `Scopy/Views/History/HistoryItemView.swift:473`, `Scopy/Views/History/HistoryItemView.swift:1003`). That is a shallow seam: the cache hides storage mechanics but leaks policy.

**Solution:** Add a `MarkdownPreviewMetricsPolicy` or deepen `MarkdownPreviewCache` with an Interface that normalizes metrics given text, raw measured metrics, font/padding/max width, and returns `MarkdownContentMetrics`. Then replace the duplicated stable-width blocks with calls to that Module. The Module can also decide when to rewrite cache metrics, keeping cache freshness policy close to normalization.

**Benefits:** Locality improves for a known fragile WebKit sizing behavior. Leverage improves because unit tests can cover under-width, non-finite, tiny width, fallback width, and max-width clamp rules once. This is a contained first step if the broader hover pipeline feels too risky.

**Risks:** This does not solve broader preview orchestration or long-frame attribution; it is mainly maintainability/stability. It should not be marketed as a performance improvement unless profile metrics change. Care must be taken not to alter popover dimensions or markdown/file-preview cache keys.

**Tests / Verification:** Add focused tests for stable metric normalization and cache update behavior. Run `make build`, `make test-unit`, `make test-strict`. For UI confidence, run `ScopyUITests/HistoryListUITests/testHoverPreviewDismissesOnScroll` and a manual/focused markdown preview check if available. Performance gate is optional unless implementation changes render timing; `make perf-frontend-profile` is prudent if touching WebKit measurement timing.

### 3. Deepen Row Action State Into A Row Action Module

**Rank:** 3. Good maintainability candidate, lower hot-path value.

**Files / Modules:** `HistoryItemView`, `HistoryItemRowController`, `HistoryItemMarkdownExportController`, image optimization callbacks in `HistoryViewModel`, `HistoryItemFileNoteEditorView`, and `HistoryItemRowControllerTests`.

**Problem:** `HistoryItemRowController` currently stores state and task handles (`Scopy/Views/History/HistoryItemRowController.swift:5`, `Scopy/Views/History/HistoryItemRowController.swift:16`) but workflows still live in `HistoryItemView`. The row view owns note editor presentation and commit (`Scopy/Views/History/HistoryItemView.swift:1288`), PNG export task lifecycle and message reset (`Scopy/Views/History/HistoryItemView.swift:1244`), image optimization task lifecycle and status message formatting (`Scopy/Views/History/HistoryItemView.swift:1399`), plus context menu command wiring (`Scopy/Views/History/HistoryItemView.swift:856`). The deletion test says deleting the controller would merely move a few stored properties back into the row; it is not yet a deep Module.

**Solution:** Deepen row actions behind an Interface such as `begin export`, `begin optimize`, `present/commit note`, and `cleanup on disappear` with injected Adapters for `onOptimizeImage`, `HistoryItemMarkdownExportController`, `onUpdateNote`, message timing, and task cancellation. Keep the context menu view in `HistoryItemView`, but move action workflow and message formatting into the action Module.

**Benefits:** Locality improves for user-visible row actions and task cleanup. Tests can validate export failure/success, message reset cancellation, note normalization, optimize message mapping, and cleanup through the action Interface instead of private view functions. This also reduces `HistoryItemView` churn without touching list virtualization.

**Risks:** It may not affect performance metrics and can be over-abstracted if it only wraps existing methods. It also touches user-visible context menu behavior, export feedback, and note editing, so regression tests are required. Keep existing strings and accessibility identifiers unchanged (`Scopy/Views/History/HistoryItemView.swift:860`, `Scopy/Views/History/HistoryItemView.swift:871`, `Scopy/Views/History/HistoryItemView.swift:883`).

**Tests / Verification:** Expand `HistoryItemRowControllerTests` or add `HistoryItemRowActionControllerTests` for export, optimize, note, cancellation, and cleanup. Run `make build`, `make test-unit`, `make test-strict`. Add focused UI tests only if context menu presentation or identifiers change; otherwise existing context menu/export tests should cover user flow.

### 4. Turn Frontend Profile Harness Into A Deeper, Typed Reporting Module

**Rank:** 4. Valuable for future performance work, but not the first product-code implementation.

**Files / Modules:** `scripts/perf-frontend-profile.sh`, `ScopyUISupport/ScrollPerformanceProfile.swift`, `ScopyUITests/HistoryListUITests.swift`, `Makefile`, and release profile docs.

**Problem:** The profile harness is effective but shallow across shell, Python, UI test, and Swift layers. The shell script owns command-line parsing, process cleanup, feature-flag variant selection, xcodebuild invocation, JSON parsing, expected scenario validation, and Markdown report formatting (`scripts/perf-frontend-profile.sh:57`, `scripts/perf-frontend-profile.sh:104`, `scripts/perf-frontend-profile.sh:121`, `scripts/perf-frontend-profile.sh:182`, `scripts/perf-frontend-profile.sh:462`, `scripts/perf-frontend-profile.sh:497`). Swift profile output owns raw frame/metric/accessibility/main-thread payloads (`ScopyUISupport/ScrollPerformanceProfile.swift:303`, `ScopyUISupport/ScrollPerformanceProfile.swift:348`, `ScopyUISupport/ScrollPerformanceProfile.swift:565`). UI tests forward variant flags manually (`ScopyUITests/HistoryListUITests.swift:13`, `ScopyUITests/HistoryListUITests.swift:321`). Adding a new metric requires synchronized edits in several places.

**Solution:** Extract the Python aggregation/reporting portion into a tested script Module with typed scenario/metric definitions and golden fixture tests. Optionally centralize the variant flag set in one shell-readable or JSON config consumed by both script and test launch logic. Keep `ScrollPerformanceProfile` raw payload stable, but make the reporting Interface own expected scenarios, bucket lists, summary tables, and evidence validation.

**Benefits:** Locality improves for metric/report churn. Leverage improves because future frontend architecture changes can add metrics with less risk of summary drift. This is important because the latest profile says missing summaries are not evidence and profile-level counters are noisy; stronger reporting prevents false confidence (`doc/perf/release-profiles/v0.7.6-profile.md:52`, `doc/perf/release-profiles/v0.7.6-profile.md:62`).

**Risks:** This is tooling work and does not directly deepen product UI Modules. It could delay the user-facing architecture improvement if selected first. Keep it out of product-code implementation unless the chosen candidate depends on adding/changing profile metrics.

**Tests / Verification:** Add fixture JSON and unit tests for the aggregator, expected-scenario failures, missing bucket behavior, and Markdown output. Run `make build`, `make test-unit` if Swift is touched, plus a profile smoke such as `make perf-frontend-profile` with short arguments to prove the summary still emits.

## Ranked Recommendation

1. **Implement Candidate 1 first: `HistoryHoverPreviewPipeline`.** It has the best Depth/Leverage/Locality ratio and directly addresses the thickest remaining frontend Module. It also aligns with the current profile reality: row descriptor and thumbnail scheduler are already done, while hover preview remains a broad source of async work, cache policy, WebKit sizing, and UI stability risk.
2. Candidate 2 is a good fallback or first slice inside Candidate 1 if the grilling loop chooses a smaller step. It reduces duplicated markdown metric policy with low blast radius.
3. Candidate 3 is worthwhile after preview work; it improves row maintainability but is less performance-sensitive.
4. Candidate 4 should be scheduled when the next implementation requires new metrics or when profile-report drift blocks confident decisions.

The recommended first implementation should be narrow: extract preview loading/planning and cache policy without changing popover rendering, UI strings, accessibility identifiers, settings behavior, or the already-deep row descriptor/thumbnail scheduler seams.

## Caveats / Not Found

- No product code was modified.
- No live profile was run during this research pass; performance conclusions rely on current checked-in v0.7.6 profile artifacts and prior logs.
- No repo `CONTEXT.md` or ADR files were found in the initial task PRD notes, so architecture vocabulary comes from `improve-codebase-architecture` and active docs.
- The frontend profile harness does not exercise hover-preview rendering in the standard real-snapshot scenarios: `hover.markdown_render_ms` and `hover.preview_image_decode_ms` are present in the summary key list but absent in the latest standard profile buckets (`scripts/perf-frontend-profile.sh:212`, `logs/perf-frontend-profile-2026-05-07_03-22-16/frontend-scroll-profile-summary.md:80`). Candidate 1 should therefore add focused preview tests and, if performance claims are made, a hover-specific profile or UI smoke.
- Replacing `List + ScrollViewReader` is intentionally not proposed. Current frontend spec says to preserve it unless measured evidence proves a better path, and the latest profile does not justify a broad list rewrite.
