# v0.7.5 Presentation Cache And Fuzzy Reuse Profile

## Scope

This profile documents the low-risk performance optimizations shipped in `v0.7.5`.

The release does not change user-facing clipboard, preview, export, selection, or settings behavior. It removes repeated presentation-time derivations and reuses bounded full-fuzzy search work that was already computed for the first page.

## Evidence Inputs

- Snapshot DB: `perf-db/clipboard.db`（6421 items / 148647936 bytes）
- Backend baseline: `logs/perf-audit-v0.7.4-current-2026-04-30_20-19-15`
- Backend current: `logs/perf-audit-perf-opt-2026-04-30_23-19-00`
- Frontend standard profile: `logs/perf-frontend-profile-2026-04-30_23-26-29/frontend-scroll-profile-summary.md`
- Unified table: `logs/perf-unified-2026-04-30_23-29-33.md`

The backend comparison uses separate audit directories against the same snapshot DB. The frontend profile is a real-scroll regression smoke: its `baseline` and `current` variants are feature-flag variants inside the same build, so it should not be read as a clean pre-change/post-change A/B for every frontend metric.

## Backend Search Highlights

| Metric | v0.7.4 baseline | v0.7.5 current | Change |
|---|---:|---:|---:|
| backend.engine.abc.force_full_fuzzy.p95_ms | 5.672ms | 0.490ms | -91.36% |
| backend.engine.cmd.force_full_fuzzy.p95_ms | 6.630ms | 0.884ms | -86.67% |
| backend.engine.full_index.peak_rss_mb | 221.766MB | 220.391MB | -0.62% |

The current run recorded `fuzzy_sorted_matches_cache_store_count=2051`, confirming that first-page full fuzzy scans populated the bounded top-K cache. This keeps the memory shape bounded while letting the next page reuse the initial scan's sorted prefix.

## Frontend Guardrails

The standard frontend profile completed all three real-snapshot scenarios and generated `logs/perf-frontend-profile-2026-04-30_23-26-29/frontend-scroll-profile-summary.md`.

Representative current p95 values:

| Scenario | Active frame p95 | Main RunLoop active p95 | Row body p95 | Display model p95 | File preview p95 |
|---|---:|---:|---:|---:|---:|
| real-snapshot-accessibility | 25.000ms | 13.749ms | 0.261ms | 0.566ms | 0.218ms |
| real-snapshot-mixed | 33.333ms | 14.966ms | 0.276ms | 0.600ms | 0.220ms |
| real-snapshot-text-bias | 25.000ms | 14.649ms | 0.250ms | 0.596ms | 0.508ms |

These numbers keep row construction and display-model work sub-millisecond in the standard profile. The mixed scenario's frame/drop counters remain noisy at the profile level, so future frontend work should keep reading main-thread and long-frame attribution rather than treating row-level buckets alone as the whole bottleneck.
