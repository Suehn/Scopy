# v0.60.2

## 📌 一页纸总结（What / Why / Result）

### What

- Fix/Clipboard：
  - 修复图片先进入 Scopy 历史、再从历史回放到 Codex 时的失败边界。
  - 标准 PNG 历史回放保持原始 `public.png` bytes；palette/indexed PNG 保留原始 `public.png`，并额外补一份 rasterized `public.tiff` fallback。
  - 临时/误分类图片文件 URL 继续按图片回放；普通 Finder 图片文件和普通文本文件继续按 file URL 回放。
- Tooling/Test：
  - `project.yml` 排除 `ScopyTests/Fixtures/**`，避免 `xcodegen` / `make build` 把 fixture 自动塞回资源阶段，保证 release 验证后的工作树干净。
  - 新增安全真实截图 fixture 与回归测试，直接锁定 Codex/macOS 剪贴板历史回放问题。

### Why

- Codex 当前 macOS 图片粘贴链路依赖 `arboard` 的窄读取路径，对某些历史 palette PNG 的回放表示不够宽容；源应用“刚复制立即粘贴”通常没问题，但同图进历史后再回放会失败。
- 不能为了兼容 Codex 就把所有 palette PNG 无差别重编码，否则会破坏普通应用看到的原始 `public.png` 表示，也会让“历史回放”和“刚复制即粘贴”差异扩大。
- release/build 流程需要在每次 `xcodegen` / `make build` 后保持工作树可预测，否则 release 验证和 tagging 很容易被无关脏改动阻断。

### Result

- 历史 palette PNG 现在保留原始 `public.png` bytes，同时给 Codex 这类窄读取链路提供一份可靠的 `public.tiff` fallback。
- 标准 PNG 历史回放继续保持原始 PNG bytes，不会只因为 Codex 而被全量改写。
- `make build`、`make test-unit`、`make test-strict`、`make docs-validate`、`make release-validate` 都已通过。

---

## 🏗️ 实现路线

1. 收紧历史图片回放策略：标准 PNG 原样回放；高风险 palette/indexed PNG 保留原始 PNG，并补一份 rasterized TIFF fallback。
2. 保留临时/误分类图片文件 URL 的图片语义，同时避免真实 Finder 文件复制被错误降级为图片复制。
3. 用真实安全截图 fixture 建立定向回归，覆盖历史图片回放到 Codex 的失败场景。
4. 修正 `project.yml`，避免 XcodeGen 把测试 fixture 重新编入资源阶段，保证构建后工作树干净。
