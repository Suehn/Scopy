# v0.64

## 📌 一页纸总结（What / Why / Result）

### What

- Perf/Search：
  - full-index warm-load 的磁盘缓存路径补上 metadata preflight、reason 归因和更轻的 payload decode，首次命中 disk cache 时延迟从约 `3049.715ms` 收敛到 `612.389ms`。
  - staged-refine 的 full-index 预热继续保留，但后台 build 改成按查询会话去重、可取消、且只在真正需要 refine 时触发。
- Refactor/Preview：
  - `HistoryItemView` 的 hover / popover / dismiss 生命周期抽到 `HistoryItemPreviewCoordinator`。
  - 修复 UITest tap-open preview 后滚动不关闭的回归。
- Infra/Observability：
  - 新增 `AsyncPermitPool` 统一文件大小计算与缩略图生成的 permit/waiter 逻辑。
  - 新增 `BestEffortFileOps`，为 cleanup / restore / pending payload 这类 best-effort 文件操作补日志上下文。

### Why

- 上一轮 perf 证据已经表明 full-index warm-load 的主要浪费来自“先重读/重校验/重解码磁盘缓存，最后又回退到 DB build”的双重付费，需要先把缓存命中与失败路径做成快失败、可解释、可量化。
- `HistoryItemView` 在同一视图里同时持有 hover debounce、popover token、preview task 和系统 dismiss observer，状态机太重，容易让滚动关闭、hover 退出和系统关闭三条路径继续分叉。
- 生产路径里若干 `try?` 型 best-effort 文件操作对用户结果语义影响不大，但对排障价值很高；继续完全静默会放大线上定位成本。

### Result

- 搜索 warm-load 体验从“接近 3 秒”降到“约 0.6 秒”，并保留现有 `Exact` / `Regex` / staged-refine / `SearchCoverage` 语义不变。
- preview 相关状态收口到 coordinator，scroll dismiss、hover exit、system dismiss 的最终状态一致性更强。
- 文件 cleanup / restore / replay 辅助路径现在能留下明确上下文，不再只剩 silent failure。

---

## 🏗️ 实现路线

1. 在 `SearchEngineImpl` 中拆出 full-index builder、disk-cache codec、warm-load metrics 与 reason 归因，先把 warm-load 成本来源量化。
2. 为 full-index 磁盘缓存增加 sidecar metadata preflight，并把 payload decode 改成更轻的 plist 手工解析，避免命中路径继续被 `Codable` decode 主导。
3. 将 preview 生命周期抽成 `HistoryItemPreviewCoordinator`，让 row view 只保留布局和交互入口。
4. 用 `AsyncPermitPool` 与 `BestEffortFileOps` 收口生产路径里重复的 permit/waiter 队列与静默文件操作。
